We are undertaking a project to access and unlock the secured diagnostic functions of the E88 Engine Control Module (ECU) on a 2019 Chevrolet Malibu by bypassing or legitimately solving the seed-key security challenge that General Motors implements via the UDS protocol (ISO 14229-1), which governs how ECUs are accessed over CAN for diagnostics, reprogramming, and advanced control; specifically, GM ECUs implement SecurityAccess (UDS Service 0x27), where an external device must request a “seed” from the ECU (e.g., by sending 0x27 0x01 or a sub-function like 0x27 0x11), receive a seed value (typically 5 bytes), use a proprietary or reverse-engineered algorithm to compute a valid “key” for that seed, and return it via 0x27 0x02 or similar to unlock a higher security level, granting access to sensitive features such as ECU reprogramming, DTC clearing, or parameter changes—functions that are otherwise locked down by the OEM; our objective is to fully reverse-engineer or simulate this seed-key unlock process using our own tools and code, enabling us to talk to the ECU directly without relying on expensive or locked-down GM tools like GDS2 or SPS; so far, we’ve gathered several key components: (1) we have a known working key generation binary (`gmkeys.exe`) that can take a 5-byte seed and return a 5-byte key for several algorithm types (e.g., Algo 87, 88, 89, etc.), and we have confirmed with known inputs that Algo 87 produces a valid key for the seed `8CE7D1FD06`, showing we can replicate key generation outside of official GM software; (2) we have a large partial dump of `STGTERM.DAT`, which appears to be a binary seed-key mapping or lookup file used by GM or third-party software for key verification or offline decoding—it’s structured as hexadecimal entries, some of which may be full seed-key pairs or transformations for specific algorithms; (3) we know from the vehicle documentation and teardown of the ECU (e.g., E88, P/N 12697426) that it supports UDS over CAN (ISO 15765-4), and uses high-speed CAN at 500kbps over OBD-II pins 6 and 14, which is compatible with our OBDLink MX+ hardware, a consumer-grade Bluetooth OBD-II adapter that supports raw CAN frame transmission and reception, GM-specific protocols like GMLAN and MS-CAN, and has been validated for similar flashing operations in tools like PCM Hammer and universal UDS libraries; (4) we’ve confirmed from other researchers and community sources that you can use custom Python, C#, or .NET applications to connect to the OBDLink MX+ via serial-over-Bluetooth, issue UDS commands like 0x10 (Diagnostic Session Control) and 0x27 (Security Access), and handle raw CAN traffic, meaning we don’t need a full J2534 device like a GM MDI or DrewTech interface for our work—only raw UDS over CAN with our seed-key logic implemented; (5) we have partial C# code and understanding of how the seed-key generation functions work, potentially involving mathematical transforms, XOR, key tables, or substitution logic depending on the selected algorithm ID, and we can port that into our own tooling or GUI app to wrap around the MX+ device; (6) we also understand the risk involved—incorrect keys may result in temporary lockout periods from the ECU or trigger anti-tamper protection, so it’s important to operate with correct seeds and only send valid computed keys when we are confident in the algorithm match; (7) we also have logs and memory captures from `gmkeys.exe` that could help us deduce algorithm behavior or map STGTERM.DAT entries to output transformations, enabling offline bruteforce, reverse-engineering, or table lookup approaches to computing valid keys; (8) we are working in C# for the tooling since our other reverse-engineering projects already involve .NET and it allows fast serial port access (for MX+ communication), structured parsing of the ECU response data, and GUI integration if needed later; (9) ultimately, our goal is to produce a full C# tool that connects to the OBDLink MX+, initiates a diagnostic session with the E88 ECU, requests the seed, uses our own implementation of the correct algorithm to compute the key (whether by calculation or table), and sends it back to unlock the ECU—confirming success through UDS positive response codes (e.g., 0x67 for success to 0x27 requests), and from there optionally dumping memory, reprogramming, or adjusting ECU parameters, giving us full control and insight into GM’s security model without reliance on locked or commercial flashing suites; this process also teaches us about low-level automotive reverse engineering, seed-key algorithm families, and how OEMs enforce firmware integrity, and lays the groundwork for more complex work like custom tuning, patching, or open-source ECU development.
